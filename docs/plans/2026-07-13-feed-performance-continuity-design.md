# 广场刷帖性能与连续性设计

## 目标

让信息流在桌面独立滚动、手机整页滚动和长时间连续刷帖三种场景下都保持稳定。页面必须继续保留较大的正文和热议预览，不能靠缩小文字或删除内容伪造“轻量”。

## 基线证据

- 首屏加载 20 张帖子卡，约 858 个 DOM 节点。
- 连续触发四次分页后增长到 56 张卡、约 1764 个 DOM 节点；右栏滚动位置仍保持为 0，证明左右栏已经独立。
- HTML、CSS、核心脚本与本地化脚本合计 gzip 约 72KB；本地首页 TTFB 约 1ms，Feed API 约 6ms。瓶颈不在网络或包体，而在长列表绘制和跨布局滚动状态。
- 从桌面深处切换为手机宽度后刷新，桌面 feed-column 的数值滚动位置会错误应用到手机 window，页面直接落在信息流中段。

## 方案比较

1. **自适应滚动状态 + 增量观察器 + J/K 浏览（采用）**：记录滚动容器模式，跨桌面/手机时不复用错误坐标；分页时只把新增卡片接入阅读观察器；桌面提供 J/K 快速切帖。改动集中，能同时改善性能与刷帖连续性。
2. **完整虚拟列表**：理论上最省 DOM，但帖子高度、展开正文、评论楼和返回锚点都高度动态，当前阶段实现风险过高。
3. **缩小正文并减少评论预览**：实现简单，却违背已经确认的可读性和“讨论有热度”目标。

## 交互

- 历史状态同时保存 `feedTop` 和 `feedScrollMode`。只有模式相同才恢复滚动位置；旧状态或跨模式状态回到顶部。
- 桌面按 J/K 在相邻帖子之间移动，焦点、右侧阅读信号和短暂高亮同步更新；输入框、弹窗和按钮不会被快捷键劫持。
- 每张帖子继续保留现有的布局、绘制和样式隔离，以确保返回、展开、自动分页和讨论锚点可靠。分页新增卡片时不再断开并重建全部阅读观察器，只观察本次新增节点。
- 减少动态效果偏好下关闭导航高亮动画。

## 验证

- 对比修复前后桌面深滚后切换到手机的起始位置。
- 连续加载完整种子 Feed，确认帖子数量、滚动高度、右栏独立性和自动分页不受影响。
- 验证 J/K、PageUp/PageDown、Home/End、输入框和弹窗键盘行为。
- 运行完整测试与 `git diff --check`。
